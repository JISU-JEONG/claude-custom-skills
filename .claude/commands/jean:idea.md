# jean:idea

사용자의 두서없는 생각을 **대화를 통해 함께 다듬어서** LLM이 효율적으로 실행할 수 있는 구조화된 명령으로 변환한다.

## 입력

`$ARGUMENTS` — 자유 형식의 생각.

---

## 핵심 원칙

<system_principles>
이 스킬의 톤·태도. 메타 수준 가이드:

1. **한 번에 결과를 내놓지 않는다** — 이해한 것을 먼저 보여주고, 모르는 것을 물어본다. 잘못된 결과보다 짧은 왕복이 효율적.
2. **사용자가 만족할 때까지 대화를 계속한다** — 몇 턴이든. 도달이 목적이지 속도가 아님.
</system_principles>

<operational_rules>
구체적 동작 룰:

3. **질문은 한 턴에 4개 이하** — `AskUserQuestion` 도구 한도. 각 질문에 "왜 묻는지·답에 따라 뭐가 달라지는지" 이유를 동반.
4. **유추 가능한 건 `[자동 판단]`으로 먼저 채운다** — 그리고 "맞나요?"로 확인. 질문은 *진짜 불확실한 것*만.
5. **사용자가 확인한 가정으로만 진행한다** — 미확인 가정으로 Step 4까지 가지 않는다 (잘못된 결과물 생산 방지).
</operational_rules>

## 실행 프로세스

```
Step 1: 초기 이해 + 컨텍스트 수집
     ▼
Step 2: 이해한 내용 확인 + 질문 (AskUserQuestion)  ◄──┐
     ▼                                                │
Step 3: 사용자 응답 반영                              │
     ▼                                                │
     충분히 명확한가? ─── No ─────────────────────────┘
     │
     Yes
     ▼
Step 4: 최종 명령 구조화 + 출력
```

### Step 1: 초기 이해 + 컨텍스트 수집

`$ARGUMENTS`를 읽고 주제를 파악한다. 동시에 유형에 맞는 컨텍스트를 수집한다.

<topic_types>
| 유형 | 신호 키워드 | 컨텍스트 수집 방법 |
|------|-------------|-------------------|
| **A. 기술 구현** | "구현", "만들", "개발", "어떻게", "방법" | CLAUDE.md + Glob/Grep로 관련 코드 + 필요시 WebSearch |
| **B. 설계 고민** | "고려", "생각해야", "체크", "선택", "결정" | 기존 아키텍처 파악 + 업계 모범사례 |
| **C. 리서치** | "뭐야", "알려줘", "차이", "비교" | WebSearch + WebFetch |
| **D. 비즈니스/기획** | "돈", "벌", "사업", "시장", "사이드 프로젝트" | WebSearch + 사용자 memory |
| **E. 문제 해결** | "에러", "안되", "이상", "깨져", "버그" | 에러 로그 + 관련 코드 + git log |
</topic_types>

복합 유형이면 가장 강한 신호를 기본으로, 나머지를 보조로 포함.

<context_budget>
시간·범위 정량 기준 (과도한 수집은 사용자 응답 시간을 낭비):

- CLAUDE.md 가 있으면 1회 Read
- Glob 검색은 패턴 1~2개, 결과는 **5개 파일 이내** 까지만 Read
- Grep 검색은 **결과 10개 이내**, 필요한 것만
- WebSearch 는 1~2회 쿼리
- 합계 도구 호출 ~10회 이내
</context_budget>

### Step 2: 이해한 내용 확인 + 질문

컨텍스트를 바탕으로 **내가 이해한 것을 먼저 텍스트로 보여준 뒤**, 불확실한 것은 `AskUserQuestion` 도구로 구조화된 질문을 던진다.

<understanding_template>
```
제가 이해한 내용입니다:
─────────────────────────
{핵심 질문을 내 말로 재정의}

확정된 것:
- {이미 결정된 사항}
- [자동 판단] {프로젝트에서 유추한 것}

파악한 컨텍스트:
- {프로젝트 스택 / 기존 구현 / 웹 조사 결과}
─────────────────────────
```
</understanding_template>

이해 확인 텍스트 다음 `AskUserQuestion` 호출. 질문 작성 규칙:

- 각 질문의 `description` 에 **왜 묻는지·답에 따라 뭐가 달라지는지** 이유 동반
- "Y/N" 대신 맥락으로 표현 ("보통은 ~인데, 여기서도 그럴까요?")
- 선택지가 명확하면 `options` 로 제시. 자유 입력이 필요하면 넓은 선택지 + `Other`
- 유추 가능한 항목은 추천 옵션에 `(Recommended)` 표시 후 첫 번째 배치
- 한 번에 최대 **4개** 질문 (도구 한도)
- 진짜 불확실한 것만 질문. 유추 가능한 건 위 텍스트의 `[자동 판단]` 으로 채워 확인만

### Step 3: 사용자 응답 반영

답변을 받으면:

1. 수정·추가 사항 반영
2. 새 정보로 컨텍스트 업데이트
3. 다음 분기 판단:
   - 핵심이 명확 + 실행 단계 / 체크리스트 작성 가능 → **Step 4**
   - 방향이 갈라지거나 사용자가 교정 중 → **Step 2 로 돌아감**
   - 사용자가 "아 그게 아니라..." → **Step 1 부터 리셋** ("알겠습니다, 다시 정리할게요.")

### Step 4: 최종 명령 구조화 + 출력

대화에서 충분히 명확해지면 유형에 맞는 형식으로 최종 명령을 출력한다.

<output_size>
출력 분량 결정 기준:

| 분류 | 줄 수 | 기준 |
|------|------|------|
| 단순 | 15~30 | 단일 파일 / 단일 함수 / 단순 리서치 / 결정 사항 1~3개 |
| 중간 | 30~60 | 여러 파일 / 다단계 절차 / 4~7개 결정 사항 |
| 복잡 | 60~100 또는 분할 | 신규 프로젝트 / 여러 도메인 / 8개 이상 결정 사항. 100줄 초과 시 phase 별로 분할해 사용자에게 다음 phase 진행 여부 확인 |
</output_size>

**공통 구조** — 모든 type 공통:
1. `## {목표 한 줄 요약}`
2. 대화에서 확인된 컨텍스트 (`[확인됨]` 태그)
3. 유형별 본문 (실행 단계 / 체크리스트 / 비교표 등)
4. "이대로 진행할까요?" 또는 동등한 마무리 질문

## Worked Examples (Type 별 1개씩)

<example_type_a>
**Type A — PDF 논문 → LLM 파이프라인**

```
사용자: /jean:idea 논문을 pdf로 받아서 llm에 넘기기 어떻게 구현해야 될까?

[Step 1: 컨텍스트 수집 — CLAUDE.md 읽기, PDF 관련 코드 탐색]

[Step 2: 이해 확인 + AskUserQuestion]
  텍스트:
    제가 이해한 내용입니다:
    ─────────────────────────
    PDF 논문을 LLM 에 전달하여 처리하는 파이프라인 구현.

    파악한 컨텍스트:
    - [자동 판단] 현재 프로젝트: Python 3.10 + Flask
    - 프로젝트 내 PDF 관련 코드: 없음 (신규)
    ─────────────────────────

  AskUserQuestion:
    질문 1: "논문 처리 규모는?" — 1편이면 단순 API, 여러 편이면 RAG 필요
      options: "1편씩 (Recommended)", "여러 편 검색", "아직 모름"
    질문 2: "어떤 정보를 뽑으려고?" — 목적에 따라 프롬프트 설계가 달라짐
      options: "요약", "QA (질의응답)", "특정 데이터 추출", "요약 + QA"
    질문 3: "이 프로젝트에 붙일까요?" — 구조가 달라짐
      options: "별도 프로젝트 (Recommended)", "기존 Flask 에 추가"

[사용자 응답: 1편씩, 요약+QA, 별도 프로젝트]

[Step 4 — 최종 출력]
## PDF 논문 → LLM 요약/QA 파이프라인 (별도 프로젝트)

### 컨텍스트
- [확인됨] 논문 1편 단위 처리
- [확인됨] 목적: 요약 + QA
- [확인됨] 별도 Python 프로젝트

### 추천 방식
Claude API 네이티브 PDF 지원 — 가장 간단하고 품질 좋음

### 실행 단계
1. 프로젝트 초기화 (Python + anthropic SDK)
2. PDF 업로드 → Claude API 전송 함수
3. 요약 프롬프트 설계
4. QA 프롬프트 설계 (논문 내용 기반 질의응답)
5. CLI 또는 간단한 웹 인터페이스

### 제약조건
- Claude 1M 컨텍스트로 대부분 논문 1편 커버
- 토큰 초과 시 섹션별 청킹 전략 필요

이대로 진행할까요?
```
</example_type_a>

```
## 로그인 (OAuth) 체크리스트

### 컨텍스트
- [확인됨] OAuth 기반, 일반 규모 (100~10K)

### 고려 영역
**1. 인증 흐름**
- [ ] OAuth provider 선택 (Google + GitHub 권장)
- [ ] redirect_uri 화이트리스트
- [ ] state 파라미터 CSRF 방지
- [ ] PKCE 적용 (모바일·SPA 시)

**2. 세션 관리**
- [ ] httpOnly·Secure 쿠키
- [ ] 세션 만료 정책 (sliding vs absolute)
- [ ] 로그아웃 시 서버측 무효화

**3. 사용자 식별**
- [ ] 이메일을 키로 쓸지 vs provider sub
- [ ] 같은 이메일의 다른 provider 처리

### 우선순위
| 순위 | 항목 | 이유 |
|------|------|------|
| 1 | state + PKCE | CSRF·인증코드 탈취 방어 — 보안 기본 |
| 2 | 세션 만료 | 장기 유지가 사용자 편의·보안 트레이드오프 |
| 3 | 동일 이메일 분기 | 흔히 놓치고 나중에 마이그레이션 어려워짐 |

### 흔한 실수
- redirect_uri 와일드카드 허용 → open redirect 취약점
- access_token 을 localStorage 저장 → XSS 노출

특정 영역을 더 깊게 파볼까요?
```

```
## RAG vs Fine-tuning 조사 결과

### 핵심 답변
RAG 는 **외부 지식을 검색해 컨텍스트에 주입**, Fine-tuning 은 **모델 가중치 자체를 도메인 데이터로 재학습**. 동적 / 빈번 갱신 데이터 → RAG, 일관된 톤·포맷·도메인 언어 학습 → Fine-tuning.

### 상세
- **RAG (Retrieval-Augmented Generation)** — 질문 시점에 관련 문서를 임베딩 검색으로 찾아 프롬프트에 함께 넣음. 모델은 그대로 둠.
- **Fine-tuning** — 학습 데이터로 모델 가중치를 추가 학습. 결과물은 새 모델 (또는 LoRA 어댑터).

### 비교표
| 항목 | RAG | Fine-tuning |
|------|-----|-------------|
| 지식 갱신 | 즉시 (DB 갱신만) | 재학습 필요 |
| 비용 | 낮음 (검색 인프라) | 높음 (GPU 학습) |
| 출처 추적 | 가능 (검색 결과 노출) | 어려움 |
| 톤/포맷 학습 | 약함 | 강함 |
| 환각 (hallucination) | 줄어듦 | 줄어들 수도/늘 수도 |

### 더 알아볼 수 있는 것
- 하이브리드 (RAG + Fine-tuning) 패턴
- RAG 의 검색 품질 측정법 (Recall@k 등)
```

```
## Docker OOM 조사 및 수정

### 컨텍스트
- [확인됨] 발생: 랜덤, 작업 패턴 미파악
- 환경: docker-compose memory 512MB
- 최근 변경: numpy·pandas 추가 (어제)

### 증상 정리
- 현상: 컨테이너 OOM kill (exit code 137)
- 빈도: 랜덤 (수 시간~며칠 간격 추정)
- 트리거: 미파악

### 조사 절차
1. `docker stats <container>` 실시간 메모리 추이 — 점진 증가면 누수 시그널
2. `docker logs --tail 100` OOM 직전 로그 패턴 — 어떤 요청에서?
3. 메모리 한계 임시 1GB 로 상향 후 재현 시도 — 단순 한계 초과인지 확인
4. Python 의 경우 `tracemalloc` 또는 `memory_profiler` 로 누수 지점

### 가능한 원인 (가능성 순)
| 순위 | 원인 | 검증 방법 |
|------|------|----------|
| 1 | numpy/pandas 큰 데이터 in-memory 처리 | 1·2 단계 로그 확인 |
| 2 | 메모리 누수 (캐시·전역 변수 누적) | `docker stats` 점진 증가 |
| 3 | 단순 한계 초과 (정상 사용량 > 512MB) | 한계 상향 후 재현 |

### 즉시 시도해 볼 것
- docker-compose memory 1GB 로 일시 상향 → 24h 관찰. 죽지 않으면 #3, 죽어도 시간 늘면 #1·#2 시그널 분리.
```
